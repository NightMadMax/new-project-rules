# ADR-0003: Двухъярусная архитектура знаний NPR и Best Practices

## Контекст

`new-project-rules` (NPR) и `Best Practices` (BP) — два совместных репозитория.
Разбор ([[docs/research/NPR_BP_KNOWLEDGE_ARCHITECTURE_2026-07-06]]) показал:

- конвейеры «harvest → кандидат → review → apply» в обоих репо структурно
  идентичны (дублирование механизма, реализованного дважды);
- клиент вынужден сам классифицировать находку и знать два механизма;
- оркестратор NPR (`promote-project-knowledge`) слеп к BP;
- гейт защиты асимметричен: BP защищён строго (protected `main`), NPR-сторона
  слабее — риск испортить базовые правила именно в NPR;
- пересечение содержимого (напр. «no secrets») — не копия, а разные по природе
  артефакты (императив vs доказанное знание), при риске дрейфа.

Требовалось решение, отталкивающееся от пользы для пользователя, а не от
механики.

## Решение

1. **Двухъярусная модель.**
   - **NPR — конституция**: малое ядро обязательных правил, которое диктуют
     администраторы; для пользователя **read-only**, версионируется сверху вниз.
     Обязателен самодостаточный baseline безопасности (не зависит от наличия BP).
   - **BP — единственный обратный поток**: весь опыт пользователей стекается
     только в BP через его review-конвейер.

2. **Две роли.** *Администратор* управляет обоими репо (авторит и версионирует
   стандарт, ревьюит кандидатов BP accept/reject, делает затвердевание BP→NPR,
   мёржит в `main`). *Пользователь* клонирует репо, ведёт проекты, потребляет
   baseline и практики, отдаёт только кандидатов в BP через PR (один файл, без
   записи в `main`). Solo носит обе шляпы. Роли различают права/режим.

3. **Правило маршрутизации по содержанию.** «Как разрабатывать» (код, стек,
   инструмент, промпт) → BP. «Структура проекта / поведение агента / оснастка» →
   NPR (только рукой администратора). Локальное/непроверенное остаётся в проекте.
   Баг инструмента NPR → обычный issue/PR (сопровождение), не knowledge-поток.

4. **managed/unmanaged на baseline проекта.** Контракт `managed`-блока
   (`<!-- new-project-rules:begin schema=N --> … :end`), уже применяемый к
   `~/.codex/AGENTS.md`, распространяется на корневой `AGENTS.md` проекта:
   baseline от NPR — внутри маркеров (обновляется миграцией), локальные правила —
   вне маркеров (не затираются). Подкаталожные scoped `AGENTS.md` остаются
   unmanaged по природе. Выбран блок-в-корне (Вариант 1), т.к. Codex авто-грузит
   один `AGENTS.md` на уровень без импорта.

5. **Судьба promotion-скиллов NPR.** `harvest-project-lessons` — удалить
   (дублирует BP-харвест, источник ушёл в BP). `promote-project-knowledge` —
   maintainer-only «затвердевание вызревшей BP-практики → правило NPR».
   `apply-promotion-candidate` — оставить, maintainer-only. Каталог
   `docs/quality/promotion-candidates/` сохраняется, но его вход — «BP-практика,
   выбранная на затвердевание», а не скан проектов.

6. **SSOT для сквозных тем (суженный Вариант C).** Только для реально сквозных
   тем (secrets, портируемость, выбор инструментов): императив — самодостаточно в
   NPR baseline; обоснование/доказательства — в BP; связь мягкой ссылкой;
   копировать rationale запрещено. Остальное разводит правило маршрутизации.

## Последствия

- **Пользователь**: один обратный путь (в BP), без классификации «в какой репо»,
  структурно не может испортить конституцию. Клиентский путь: harvest → PR в BP →
  pull + `apply-best-practices`.
- **Дедупликация**: обратный поток остаётся один; клиентский promotion-пайплайн
  NPR сворачивается; в NPR остаётся тонкий maintainer-инструмент внесения
  изменений. Репозитории **не сливаются**.
- **Совместимость с локальным накоплением**: top-down обновление переписывает
  только managed-блок; локальные правила и знание пользователя сохраняются;
  существующие проекты адаптируются через `standardize`/миграцию
  (`unmanaged_conflict`).
- **План внедрения (фазы):** Ф1 — managed/unmanaged на baseline проекта; Ф2 —
  свернуть/перенастроить три скилла; Ф3 — routing-правило + SSOT (+
  `GLOBAL_AGENT_INSTRUCTIONS`). После каждой code-фазы: полное код-ревью и тесты.
  Ф4 (документация, два трека: пользователь и администратор) — отложена.
- **Версионирование**: изменение состава скиллов и шаблона требует bump
  `STANDARD_VERSION`, запись в `CHANGELOG.md` и migration-entry.
- **Затрагиваемые артефакты**: `AGENTS.template.md`, `bootstrap-new-project.*`,
  `plan-migration`/`validate`, `sync_global_agents.py` (переиспользование),
  каталог `.agents/skills` и мосты `.claude/skills`,
  `promotion-candidates/README`, `USE_THIS_PROJECT` §8,
  `AI_KNOWLEDGE_PORTABILITY`, `GLOBAL_AGENT_INSTRUCTIONS`.

## Отклонённые альтернативы

- **Слить репозитории или пайплайны** — теряется разное управление и review;
  разделение бэкендов пользователю невидимо и правильно.
- **Оставить два клиентских пайплайна + маршрутизация** — слабее: не устраняет
  дублирование и оставляет классификацию на пользователе.
- **Managed через разделение по файлам (Вариант 2)** — Codex не авто-грузит
  второй файл на уровне корня; проектно-глобальные локальные правила ломают
  Codex/Claude-паритет.
- **SSOT Вариант A/B (канон целиком в одной базе)** — не отражает разную природу
  «императив vs знание»; A вдобавок зависит от наличия BP.
