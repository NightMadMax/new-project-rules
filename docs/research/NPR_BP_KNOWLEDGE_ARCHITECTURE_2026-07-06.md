# Research: архитектура знаний между new-project-rules и Best Practices

## Question

У нас два совместных репозитория — `new-project-rules` (NPR) и `Best Practices`
(BP). Оба извлекают уроки из проектов, ведут кандидатов и возвращают знание в
проекты. Где здесь настоящее дублирование, а где осознанное разделение, и как
должно выглядеть решение, если отталкиваться от **пользовательского пути**, а не
от механики?

Решение (ADR) намеренно отложено: цель заметки — детальный разбор и варианты.

## Context: что делает каждый проект

**new-project-rules (NPR) — «как начинать и вести любой проект одинаково».**
Оснастка и мета-стандарт: bootstrap, правила агента (`AGENTS.md`), шаблоны,
валидатор, doctor, миграции. Отдельно — умеет улучшать сам себя через promotion:
`harvest-project-lessons → promote-project-knowledge → apply-promotion-candidate`,
кандидаты в `docs/quality/promotion-candidates/`. Цель применения — **сам
стандарт**.

**Best Practices (BP) — «что мы поняли про то, как хорошо строить».**
Курируемая библиотека инженерного знания по разделам (`common`, `web`, `1c`,
`tools`, `anti-patterns`, `prompts`, `snippets`). Конвейер:
`harvest-practice-candidates → review-practice-candidates → apply-best-practices`,
кандидаты в `candidates/`, принятые — в `practices/<раздел>/`. Цель применения —
**пользовательский проект**.

## Finding 1. Конвейеры структурно идентичны

| Этап | NPR | BP |
|---|---|---|
| Собрать из проекта | `harvest-project-lessons` | `harvest-practice-candidates` |
| Разобрать / принять | `promote-project-knowledge` | `review-practice-candidates` |
| Применить | `apply-promotion-candidate` | `apply-best-practices` |
| Формат кандидата | `PC-<год>-<hex>`, один файл | `PC-<год>-<hex>`, один файл |

Соглашение о кандидатах уже де-факто общее: NPR сам родил кандидатов
`collision-resistant-ids` и `one-file-per-candidate`, которыми пользуется BP.
То есть паттерн один, реализован дважды (разные скрипты: `promotion_candidates.py`
против `new_candidate.py`), в двух репозиториях с разным режимом ревью.

## Finding 2. Дублирование чувствуется на «захвате», а не на «применении»

Асимметрия применения: `apply-best-practices` вливает знание **в проект
пользователя**, а `apply-promotion-candidate` меняет **саму оснастку**. Поэтому
для человека, строящего обычный проект, «получить знание в проект» — это только
BP. Двоение ощущается лишь на шаге **«я что-то узнал»**, где у пользователя три
почти одинаковые двери и он должен угадать нужную:

- оставить локально (`reflect-and-record` → DEFECTS / PLAYBOOK / AGENTS);
- отправить в BP (инженерный приём);
- отправить в стандарт NPR (структура / поведение агента / оснастка).

При этом оркестратор NPR (`promote-project-knowledge`), который **уже умеет**
выбирать маршрут, **слеп к BP**: он маршрутизирует только внутри стандарта. Это и
есть настоящая дыра — не «лишний пайплайн», а **неунифицированный вход и
BP-неосведомлённый оркестратор**.

## Finding 3. Пересечение содержимого — разные артефакты, не копия

Конкретная коллизия (единственная принятая практика BP на сегодня):

- **NPR**, `AGENTS.md`: императив «Never commit secrets, tokens, private keys,
  passwords, or real credentials.» — короткое, всегда включённое, применяется к
  каждому проекту автоматически.
- **BP**, `practices/common/PC-2026-001`: практика «Хранить credentials вне
  репозитория» — контекст, проблема, решение, проверка, `evidence_level: E2`,
  провенанс (`jira-analytics`, `router-watchdog-ops`), `review_by`, границы
  `applies_to` / `does_not_apply_to`.

Вывод: на одну тему это **два разных по природе артефакта**, а не дубль:

- NPR держит **императив** — «агент обязан / не должен», принудительно и всегда;
- BP держит **знание** — почему, доказательства, как проверить, где применимо.

Риск — не в самом факте пересечения, а в **дрейфе**: если формулировки скопированы
и одна меняется без другой.

## Finding 4. Жёсткое ограничение: AGENTS должен быть самодостаточным

Модель BP — pull: база может быть **не склонирована** на машине (та же проблема
«курицы и яйца», что уже отмечена в
[[docs/research/BEST_PRACTICES_INTEGRATION]]). Значит правило в `AGENTS.md`
**нельзя** свести к «см. практику BP X» — при отсутствии базы правило исчезнет.
Императив обязан оставаться самодостаточным в NPR; на BP можно ссылаться мягко
(за обоснованием), но не как на жёсткую зависимость.

## Finding 5. Настоящих сквозных тем мало

Проверка гипотезы «императив → NPR, знание → BP» на фактических темах-кандидатах
(из `AGENTS.md` NPR и разделов BP):

| Тема | Императив (NPR) | Знание (BP) | Разделение работает | Куда |
|---|---|---|---|---|
| Secrets вне репозитория | да | да (evidence) | да | обе, по Варианту C |
| Портируемость скриптов macOS/Windows | да | да (техника) | да | обе, по Варианту C |
| Выбор инструментов / tech-radar | да (политика) | да (радар `tools/`) | да | обе, по Варианту C |
| Не два агента в одной рабочей копии | да | почти нет | нет | только NPR |
| Дисциплина defect tracking | да (задаёт файл) | почти нет | нет | только NPR |
| Дисциплина pattern playbook | да (задаёт файл) | почти нет | нет | только NPR |
| Машиночитаемое первично | да | почти нет | нет | только NPR |
| Промпт / skill-паттерны | rule-authoring | да (`prompts/`) | это разное | раздельно |

Вывод: **настоящее пересечение узкое** — по сути три темы (secrets,
портируемость, выбор инструментов), где реально есть и «императив», и
«доказанное знание». Всё остальное — либо **чисто NPR** (дисциплина/оснастка
стандарта, у которой нет «знаниевой» половины, и в BP ей не место), либо
**естественно раздельное** (rule-authoring ≠ библиотека промптов). Значит
правило маршрутизации по содержанию закрывает большинство случаев, а SSOT-модель
нужна только для узкого набора.

## Архитектурное направление (из пользовательского пути)

1. **Не сливать репозитории и не сливать пайплайны.** Разные адресаты и разные
   режимы ревью (BP — многоконтрибьюторный с `CODEOWNERS`; NPR — личный
   стандарт). Разделение бэкендов пользователю невидимо и правильно.
2. **Одна входная дверь + маршрутизация.** У пользователя один жест — «я узнал
   полезное, сохрани куда надо» (`reflect-and-record`), а решение «локально / в
   BP / в стандарт» принимает **правило маршрутизации по содержанию**:
   - как **разрабатывать** (код / стек / инструмент) → BP;
   - как устроен **проект** / ведёт себя **агент** / оснастка → NPR;
   - локальное или непроверенное → остаётся в проекте.
3. **BP-aware оркестратор.** Научить `promote-project-knowledge` уметь передать
   кандидата в BP-конвейер, а не только в NPR. Недостающий мостик.
4. **Сквозные правила — вопрос открыт (см. ниже).**

## Открытый вопрос: канон сквозных правил (SSOT)

Тема «no secrets» показывает общий класс правил, живущих на границе. Варианты:

- **Вариант A — канон в BP `common`.** Полное знание с доказательствами — в BP;
  в `AGENTS.md` остаётся самодостаточный императив + мягкая ссылка. Плюс: знание
  в библиотеке знаний. Минус: важное правило зависит от наличия базы, если не
  оставить дубль-императив (а его оставить придётся — Finding 4).
- **Вариант B — канон в `AGENTS.md` NPR.** Правило всегда присутствует и
  применяется; BP-практика ссылается на него. Плюс: гарантированное применение.
  Минус: BP хочет самодостаточные практики с доказательствами, ссылка «наружу»
  ослабляет библиотеку.
- **Вариант C — разделение по природе (не по теме).** Императив «обязан / не
  должен» → всегда в `AGENTS.md` NPR (самодостаточно); объяснение «как хорошо,
  доказательства, нюансы» → BP. На одну тему сознательно допускаются **два
  разных артефакта**, но **копирование обоснования запрещено**: rationale живёт
  в BP, императив — в NPR, связь мягкой ссылкой. Плюс: отражает реальную разницу
  ролей, снимает дрейф текста. Минус: нужно уметь отличать «императив» от
  «знания» и держать дисциплину ссылок.

С учётом Finding 5 наклон — к **Варианту C**, но **суженному**: разделение
«императив → NPR / знание → BP» применяется только к узкому набору настоящих
сквозных тем (secrets, портируемость, выбор инструментов), с запретом копировать
обоснование. Остальное разводится правилом маршрутизации: дисциплина и оснастка
стандарта остаются в NPR, библиотека промптов/сниппетов — в BP.

## Развитие направления: двухъярусная модель (под обсуждением)

Обсуждается более сильное упрощение, чем «две двери + маршрутизация»:

- **NPR — конституция.** Малое ядро обязательных правил, которое диктуют
  мейнтейнеры; для клиента **read-only** (потребляет, не меняет), но
  версионируется нами сверху вниз. Обязателен самодостаточный baseline
  безопасности (Finding 4).
- **BP — единственный обратный поток.** Весь опыт клиентов (1С, Web, промпты,
  приёмы работы с агентом) стекается только в BP через его review-конвейер.

Что упрощает: убирает клиентский промоушен-пайплайн NPR, снимает с клиента
классификацию «в какой репозиторий», защищает базовые правила (клиент их не
трогает), делает клиентский путь одной дверью (harvest → PR в BP → pull + apply).

Отдельно разводятся два канала: **опыт/знание → только BP**; **починка
инструмента NPR (баг в скрипте/шаблоне) → обычный PR к NPR**. «Рекомендация BP →
обязательное правило NPR» остаётся, но как редкий **maintainer-only** акт
затвердевания, не клиентский.

### Уже существует (не забыть при проектировании)

В NPR **уже есть** скиллы, реализующие обратный поток «из других проектов в
корень/стандарт»: `harvest-project-lessons` (сканирует соседние git-проекты →
кандидаты в `docs/quality/promotion-candidates/`), `promote-project-knowledge`
(оркестрация) и `apply-promotion-candidate` (внесение в `rules/templates/tests/
validators/scripts/guides/skills`). Именно этот клиентский backward-in-NPR поток
двухъярусная модель предлагает **свернуть или перевести в maintainer-only**. При
проектировании ADR это надо не забыть: решение затрагивает судьбу этих трёх
скиллов и каталога `promotion-candidates/`, а не только BP.

### Судьба трёх promotion-скиллов NPR (согласованное направление)

При двухъярусной модели клиентский обратный поток в NPR закрывается, поэтому:

| Скилл | Судьба | Почему |
|---|---|---|
| `harvest-project-lessons` | **удалить** | дублирует BP `harvest-practice-candidates`; источник уроков ушёл в BP |
| `promote-project-knowledge` | **maintainer-only** | сузить/переименовать под редкий акт «вызревшая BP-практика → правило NPR»; клиентская маршрутизация обнулилась |
| `apply-promotion-candidate` | **оставить, maintainer-only** | контролируемое внесение решённого изменения в артефакты стандарта |

Каталог `docs/quality/promotion-candidates/` сохраняется, но меняет вход:
кандидаты приходят из «BP-практика, выбранная на затвердевание», а не из скана
проектов. Стандарт-уровневые идеи клиента идут обычным issue/PR к NPR (канал
починки/фичи инструмента). Затрагивает клиентские тексты: `USE_THIS_PROJECT`
раздел 8, `AI_KNOWLEDGE_PORTABILITY`, описания скиллов — переписать под «делись
через BP».

### Конфликт с локальным накоплением и managed/unmanaged

Вопрос: не конфликтует ли «неизменная сверху» конституция с изменениями, которые
постепенно копятся у пользователей локально?

- **BP конфликтовать не может** — не копируется, а тянется свежей при
  `apply-best-practices`; локальной копии нет. Весь риск — только на стороне NPR
  (копируется в проект и на машину).
- Локально копятся две разные вещи: **локальные правила** (правки поведения
  агента в своём проекте) — вот тут возможен конфликт с top-down обновлением; и
  **локальное знание** (`DEFECTS`/`PLAYBOOK`/находки) — NPR им не управляет, оно
  живёт локально до отправки в BP, конфликта нет.
- Механизм решения **уже реализован для глобальных правил**: модель
  `managed / unmanaged` в `sync_global_agents.py` — managed-блок (маркеры +
  sha256) принадлежит стандарту, внешний текст пользователя сохраняется без
  изменений; статус `unmanaged_conflict` дописывает managed-блок ниже локальных
  строк; всё через fingerprint-миграцию (`--plan`/`--apply`), никогда не молча.

**Зазор:** на уровне **проекта** `AGENTS.md` кладётся простой копией, без
разделения managed/unmanaged. Требование к архитектуре: распространить тот же
контракт на baseline проекта — корневой baseline = managed (обновляется
миграцией), локальные правила → unmanaged-слой (scoped/nested `AGENTS.md`),
который апдейт не трогает; существующие проекты обновляются через
`standardize`/миграцию с планом.

**Развилка дизайна (думаем):** managed-блок с маркерами **внутри** корневого
`AGENTS.md` (прощает правки в корне) против разделения **по файлам** (корень =
managed, локальное — только в scoped-файлах; строже и проще).

**Вывод:** двухъярусная модель не конфликтует с локальным накоплением при
условии, что managed/unmanaged-контракт распространён с глобального уровня на
baseline проекта.

## Две роли: администраторы и пользователи

Двухъярусная модель проявляется как две роли (различают права/режим, не
обязательно разных людей — solo носит обе шляпы).

**Администратор** — управляет репозиториями NPR и BP: авторит и версионирует
стандарт NPR (baseline `AGENTS`, шаблоны, скрипты, валидаторы, `STANDARD_VERSION`,
миграции); ревьюит кандидатов BP (`review-practice-candidates`, CODEOWNERS,
branch protection) — accept/reject; делает затвердевание BP-практики → правило
NPR; имеет merge в `main` обоих репо.

**Пользователь** — клонирует репозитории и доставляет кандидатов: настраивает
машину, создаёт и ведёт проекты; потребляет baseline (bootstrap/миграция) и
практики (`apply-best-practices`); отдаёт только кандидатов в BP через PR (один
файл, без записи в `main`); локальные находки держит у себя; баг инструмента NPR
— обычным issue/PR, но стандартом не управляет.

| | Пользователь | Администратор |
|---|---|---|
| Запись в `main` (NPR/BP) | нет (только PR-кандидат в BP) | да (через ревью) |
| NPR-скиллы | `setup-new-computer`, `create-new-project`, `assess`/`standardize`/`compress`, `reflect-and-record`, validate/doctor | + `promote-project-knowledge` (затвердевание), `apply-promotion-candidate` |
| BP-скиллы | `harvest-practice-candidates`, `apply-best-practices` | + `review-practice-candidates` |
| Удаляется | — | `harvest-project-lessons` |

Деление BP-скиллов чистое: harvest + apply → пользователь, review →
администратор. Документация в будущем разводится на два трека — руководство
пользователя и руководство администратора.
