# Руководство пользователя

Это руководство — простое, «человеческое» объяснение, как пользоваться этим
набором правил в повседневной работе. Без внутренней механики: только что нажать,
что при этом произойдёт и зачем это вам.

Если нужна точная справка по фразам и командам — смотрите
[[docs/guides/USE_THIS_PROJECT|«Как работать с этим проектом»]] и
[[docs/guides/MANUAL_SCRIPTS|«Ручной запуск скриптов»]]. Это руководство —
верхний, самый простой слой над ними.

> Почти всё ниже можно сделать **двумя способами**: просто **попросить
> ИИ-агента** словами или **запустить скрипт вручную**. Оба способа приводят к
> одному результату — выбирайте, что удобнее.

---

## 1. Что это за проект

Это не программа и не приложение. Это **готовый набор правил, шаблонов и
помощников**, с которого удобно начинать любой новый проект.

Обычно каждый новый проект начинается с одного и того же беспорядка: где хранить
заметки, как назвать папки, что писать в инструкциях для ИИ, что не закоммитить
случайно. Этот набор закрывает такие вопросы раз и навсегда.

**Какие задачи он решает:**

- быстро **создать новый проект** с уже готовой, аккуратной структурой;
- **настроить новый компьютер** под эту работу за один заход;
- **разобраться со старым проектом** — оценить его и по желанию привести к
  порядку;
- **проверить**, что в проекте всё на месте;
- **навести чистоту**, когда проект за долгую работу оброс лишним;
- **сохранять удачные приёмы и уроки**, чтобы не наступать на одни грабли дважды.

**Чем это помогает вам на практике:**

- меньше рутины на старте — не нужно каждый раз придумывать структуру заново;
- во всех проектах **единый порядок**, в котором легко ориентироваться;
- ИИ-ассистенты (Codex и Claude Code) **с первой минуты работают по вашим
  правилам**, а не наугад;
- ничего важного не забывается, потому что нужные файлы создаются сразу.

Всё это живёт внутри одной общей папки-хранилища (Obsidian **vault**): каждый
проект — отдельная папка, отдельный Git-репозиторий и отдельный проект на GitHub.

## 2. Установка и Obsidian vault

Чтобы использовать стандарт на новом компьютере, сначала получите репозиторий
и разместите его внутри общей папки Obsidian vault.

Потребуются Obsidian, Git и доступ к GitHub. Сам стандарт ничего не
устанавливает без разрешения пользователя.

**Через агента:**

> Склонируй `NightMadMax/new-project-rules` в общую папку моих проектов.
> Ничего не устанавливай без разрешения.

**Вручную:**

```sh
git clone git@github.com:NightMadMax/new-project-rules.git
```

Если SSH ещё не настроен:

```sh
git clone https://github.com/NightMadMax/new-project-rules.git
```

В Obsidian откройте **общую родительскую папку** как vault, а не каталог
`new-project-rules` и не отдельные папки проектов. Каждый проект остаётся
самостоятельным Git-репозиторием внутри общего vault; отдельная `.obsidian` и
вторая копия Markdown не создаются.

Подробнее: [[docs/guides/SETUP_NEW_COMPUTER|SETUP_NEW_COMPUTER]].

## 3. Быстрый старт: с чего начать

Если вы здесь впервые, пройдите три шага по порядку:

1. **Настройте компьютер** — один раз на каждой машине (см. раздел 5).
2. **Создайте первый проект** — выберите «размер» и получите готовый каркас
   (см. раздел 6.1).
3. **Откройте общую папку (vault) в Obsidian** — и работайте: заметки, код и
   документация уже связаны между собой.

Самый простой вариант каждого шага — **попросить ИИ-агента словами**. Если
предпочитаете команды — они есть в каждом разделе ниже и в
[[docs/guides/MANUAL_SCRIPTS|MANUAL_SCRIPTS]].

## 4. Как разговаривать с агентом

С агентом можно общаться обычным языком. Чтобы результат был предсказуемым,
держитесь простого правила: **скажите, что нужно сделать, и с чем именно**
(какой проект или папка).

Пример хорошего запроса:

> Создай новый проект «Мои заметки» вот в этой папке, небольшой. Сначала покажи
> план.

### Слова безопасности

Это короткие фразы, которые держат агента в рамках. Добавляйте их, когда хотите
контроля:

- **«Сначала покажи план»** — агент опишет, что собирается сделать, до того как
  что-то менять.
- **«Только посмотри, ничего не меняй»** — безопасный режим для проверки и
  оценки.
- **«Ничего не удаляй без моего согласия»** — особенно для старых проектов.
- **«Не устанавливай ничего без разрешения»** — чтобы на компьютер не попало
  лишнее.
- **«Сделай и синхронизируй»** / **«Не выкладывай в GitHub»** — управляете тем,
  уходят ли изменения в облако.

Не бойтесь ошибиться в формулировке: если запрос слишком короткий («сделай
проект»), агент переспросит детали.

## 5. Настройка нового компьютера

Делается **один раз** на каждой машине, прежде чем создавать проекты.

**Через ИИ:** попросите словами или используйте команду скилла.

- Claude Code: `/setup-new-computer`
- Codex: `$setup-new-computer`

> Настрой этот компьютер для работы с моими проектами. Ничего не устанавливай
> без моего разрешения, сначала покажи, чего не хватает.

**Вручную скриптом** (macOS/Linux; на Windows — те же файлы с расширением
`.ps1`):

```sh
./scripts/setup-global-agents.sh          # настроить общие правила для ИИ
./scripts/check-environment.sh both       # проверить, всё ли готово
```

**Что при этом происходит на компьютере:**

- создаются **общие правила для ИИ-агентов** — файлы `~/.codex/AGENTS.md` и
  `~/.claude/CLAUDE.md`, чтобы Codex и Claude Code читали одни и те же правила;
- ваши уже существующие настройки **не затираются** — скрипт аккуратный и
  безопасен при повторном запуске;
- проверяется, что на месте нужное окружение (Git, доступ к GitHub и т.д.), и
  агент подсказывает, чего не хватает.

Больше деталей: [[docs/guides/SETUP_NEW_COMPUTER|SETUP_NEW_COMPUTER]].

## 6. Работа с проектами

Главный раздел. Здесь — всё, что вы делаете с проектами: создать новый, разобрать
старый, проверить и почистить.

### 6.1. Создать новый проект

**Через ИИ:**

- Claude Code: `/create-new-project`
- Codex: `$create-new-project`
- Или просто словами:

> Создай новый проект «Название» вот в этой папке. Сначала покажи план файлов.

**Вручную скриптом:**

```sh
./scripts/bootstrap-new-project.sh "/путь/к/Новый проект" "Новый проект" software
```

**Какой «размер» проекта выбрать.** При создании выбирается профиль — это просто
масштаб будущего проекта:

| Профиль | Простыми словами |
|---|---|
| `minimal` | Маленькая заметка или эксперимент — только самое необходимое. |
| `software` | Настоящее приложение — плюс архитектура, тесты, история изменений. |
| `operated` | То, что работает «в проде» — плюс инструменты, интеграции, окружения. |
| `all` | Максимум — плюс API, модель данных и безопасность. |

Не уверены — берите `software`; при желании потом можно достроить.

**Что вы получаете на выходе:** готовый каркас проекта — правильные папки,
базовые документы (`README`, `INDEX`, `PROJECT` и др.), правила для ИИ и, при
необходимости, git/GitHub. Ничего не нужно собирать вручную.

**Чем это лучше, чем начинать с нуля:**

- не приходится вспоминать, какие файлы вообще нужны, — они уже на месте;
- одинаковая структура во всех проектах, в них легко возвращаться;
- ИИ сразу понимает проект и работает по правилам;
- меньше ошибок и «мусора» в самом начале.

Подробнее: [[docs/guides/CREATE_NEW_PROJECT|CREATE_NEW_PROJECT]].

### 6.2. Оценить существующий проект

Когда проект уже есть, а вы хотите **сначала посмотреть, не меняя ничего**:
насколько он близок к порядку и что мешает.

**Через ИИ:** `/assess-existing-project` (Codex: `$assess-existing-project`)

> Посмотри этот проект и скажи, можно ли безопасно привести его к стандарту.
> Только оценка, ничего не меняй.

Агент вернёт понятный отчёт: что уже хорошо, чего не хватает, стоит ли наводить
порядок прямо здесь или лучше собрать проект заново.

Подробнее: [[docs/guides/ASSESS_EXISTING_PROJECT|ASSESS_EXISTING_PROJECT]].

### 6.3. Привести существующий проект к порядку

Когда решение принято и нужно **аккуратно стандартизировать** старый проект.

**Через ИИ:** `/standardize-existing-project`

> Приведи этот проект к порядку. Ничего не удаляй без согласования, сначала
> покажи план.

**Вручную скриптом:**

```sh
./scripts/standardize-existing-project.sh --root "/путь/к/Проекту"
```

Работа идёт **осторожно и по шагам**: агент предложит план и не будет удалять
ваши файлы без разрешения.

Подробнее:
[[docs/guides/STANDARDIZE_EXISTING_PROJECT|STANDARDIZE_EXISTING_PROJECT]].

### 6.4. Проверить проект

Быстрая проверка «всё ли на месте» — структура, документы, окружение.

**Через ИИ:**

> Проверь этот проект и коротко скажи, что не так. Ничего не меняй.

**Вручную скриптом:**

```sh
./scripts/validate-project.sh --root . --kind project --profile software
./scripts/project-doctor.sh --root . --report-only     # диагностика среды
```

Это безопасно: проверка только смотрит и сообщает, но ничего не меняет.

Подробнее: [[docs/guides/VALIDATE_AND_DIAGNOSE|VALIDATE_AND_DIAGNOSE]].

### 6.5. Почистить проект

Когда за долгую работу накопились лишние журналы, черновики и «мусор».

**Через ИИ:** `/compress-project`

> Проект оброс — покажи, что можно безопасно убрать.

**Вручную скриптом:**

```sh
./scripts/compress-project.sh --root .            # только отчёт, ничего не трогает
./scripts/compress-project.sh --root . --apply    # применить
```

Важное свойство: чистка **архивирует, а не удаляет** — ничего ценного не теряется.

Подробнее: [[docs/guides/COMPRESS_PROJECT|COMPRESS_PROJECT]].

## 7. Сопутствующий репозиторий Best Practices

Рядом со стандартом живёт отдельная база — **Best Practices**. Это копилка
проверенных решений «как делать хорошо»: общие приёмы и приёмы под конкретный
стек (например, веб или 1С).

Идея простая: новый проект получает не только пустой каркас, но и **готовый
полезный опыт**.

**Как подключить — автоматически:**

- на последнем шаге создания проекта агент **предложит** подтянуть базу и сразу
  применит общие практики (`common`);
- когда определится стек проекта, агент **напомнит** подтянуть практики под этот
  стек;
- если базы ещё нет на компьютере — предложит её склонировать;
- ваш ответ запоминается, чтобы **не спрашивать одно и то же повторно**.

**Как подключить — вручную:**

- склонируйте репозиторий `Best Practices` рядом с общей папкой (как соседнюю
  папку);
- при создании или доработке проекта попросите агента применить практики:

> Подтяни общие практики из базы Best Practices в этот проект.

> Стек определился — примени практики Best Practices под него.

**Что это даёт:** проекты становятся не просто одинаковыми по структуре, а ещё и
качественными по содержанию — с самого начала.

## 8. Сохранение находок и уроков

Проект умеет **учиться** на вашей работе, чтобы одни и те же грабли и удачные
приёмы не приходилось открывать заново.

- **Нашли удачный приём** (сработал не один раз) или получили проверенное
  решение после перебора нескольких неудачных вариантов — он автоматически
  записывается в «playbook» проекта, чтобы повторять известное-хорошее.
- **Напоролись на проблему или ошиблись** — это фиксируется в списке дефектов
  проекта, чтобы не наступить снова.
- **Агент ошибся или вы его поправили** — попросите записать урок:

  - Claude Code: `/reflect-and-record` (Codex: `$reflect-and-record`)

  > Ты только что ошибся вот в чём: … Разберись в причине и запиши урок туда, где
  > ему место.

Если урок оказался полезен **не только для этого проекта**, пользователь
передаёт его кандидатом в **Best Practices**. Обязательный стандарт
`new-project-rules` меняют только его администраторы после проверки зрелой
практики.

> Посмотри, есть ли здесь приём, который стоит предложить в Best Practices.

Подробнее: [[docs/guides/AI_KNOWLEDGE_PORTABILITY|AI_KNOWLEDGE_PORTABILITY]].

## 9. Best Practices и сохранение находок — в чём разница

Это два разных механизма, и их легко перепутать. Разница — в **направлении
движения опыта**.

- **Best Practices** — это **вход**: готовый, заранее собранный опыт «как делать
  хорошо» **втекает в ваш проект** (общие приёмы + приёмы под стек). Вы
  *получаете* чужой и накопленный опыт.
- **Сохранение находок** — это **выход**: опыт рождается **из вашей работы**.
  Сначала он остаётся в конкретном проекте, а по желанию передаётся кандидатом
  в **Best Practices**. Вы *отдаёте* свой опыт дальше, не изменяя обязательный
  стандарт напрямую.

| | Best Practices | Сохранение находок |
|---|---|---|
| Откуда берётся | Заранее собранная соседняя база | Ваша работа в проекте |
| Куда движется | **В** проект | **Из** проекта (→ кандидат в Best Practices) |
| Зачем | Начать сразу с хороших решений | Не повторять ошибки и удачи заново |
| Когда срабатывает | При создании/доработке проекта | После находки, урока или поправки |

Проще всего запомнить так: **Best Practices приносит опыт вам, а сохранение
находок уносит ваш опыт дальше** — сначала в проект, затем кандидатом в Best
Practices.

## 10. Синхронизация и перенос между компьютерами

Работа не привязана к одному компьютеру.

- Каждый проект — это Git-репозиторий, связанный с **GitHub**. После изменений
  агент (или вы командой `git`) отправляет их в облако.
- Чтобы продолжить на другой машине: сначала **настройте её** (раздел 5), затем
  создайте общую папку, склонируйте внутрь нужные проекты и откройте эту папку
  как vault в Obsidian.

Полезные фразы:

> Сделай и синхронизируй с GitHub. — чтобы изменения ушли в облако.

> Не выкладывай в GitHub. — чтобы изменения остались только на этой машине.

Так «ничего не теряется»: главный экземпляр всегда есть в GitHub, а компьютеров
может быть сколько угодно.

## 11. Поддержание правил в актуальном состоянии

Со временем сами правила и шаблоны улучшаются. Чтобы оставаться на актуальной
версии:

> Проверь, не разошлись ли общие правила ИИ на этом компьютере с текущими.

> Обнови правила безопасно: сначала покажи, что именно поменяется.

Обновление всегда **сначала показывает изменения** и применяется только после
вашего согласия — ничего не меняется втихую.

## 12. Словарь терминов

- **vault (хранилище)** — общая папка, которую вы открываете в Obsidian. Внутри
  неё лежат все проекты.
- **проект** — отдельная папка внутри vault; одновременно Git-репозиторий и
  проект на GitHub.
- **профиль** — «размер» проекта при создании: `minimal`, `software`,
  `operated`, `all`.
- **агент** — ИИ-ассистент (Codex или Claude Code), который выполняет ваши
  просьбы.
- **скилл** — готовый сценарий для агента, вызывается как `/имя` (Claude Code)
  или `$имя` (Codex).
- **bootstrap** — создание нового проекта из шаблона «одной командой».
- **репозиторий (repo)** — папка под управлением Git; хранит историю изменений.
- **GitHub** — облако, где хранится главная копия репозитория.

## 13. Если что-то пошло не так (FAQ)

**Агент сделал не то, что я хотел.**
Попросите откатить или уточните задачу. На будущее добавляйте «сначала покажи
план» — тогда вы увидите намерение до изменений.

**Не уверен, безопасно ли выполнять действие.**
Добавьте «только посмотри, ничего не меняй». Оценка и проверка ничего не портят.

**Кажется, в проекте что-то не так.**
Попросите проверку: «проверь проект и скажи, что не так» — или запустите
`project-doctor` (раздел 6.4).

**Не помню название скилла.**
Не нужно. Пишите обычными словами — агент сам подберёт нужный сценарий.

**Боюсь потерять работу.**
Регулярно синхронизируйтесь с GitHub (раздел 10). Чистка проекта архивирует, а
не удаляет (раздел 6.5).
