# Работа с проектами — карта действий пользователя

Что можно делать с проектами по стандарту. Каждое действие расписано одинаково:
что делает, когда, где, два маршрута (🗣 агенту / ⌨ вручную) и результат.
Визуальная версия: [projects-user.html](assets/projects-user.html).

**Почему не один сквозной workflow.** Линейная модель «шаг за шагом» подходит
только для процессов (создание проекта). «Оценить», «стандартизировать»,
«сжать» — самостоятельные действия. Почти всё запускается из папки
`Правила для нового проекта`; действия над своим проектом — из его папки.

## Создать новый проект

- **Что делает:** разворачивает новый проект по стандарту — отдельный
  git-репозиторий внутри общего vault, стартовые документы, отдельный GitHub-репозиторий.
- **Когда:** начинаешь новый проект.
- **Где:** `Правила для нового проекта` (bootstrap использует её commit как
  проверяемый `source_commit`).
- 🗣 **Агенту:** *«Создай новый проект „Название“ в папке `<путь>`, профиль
  software, стек `web` и `backend`, приватный GitHub-репозиторий»* ·
  `/create-new-project`. Профиль по умолчанию `software` (есть `minimal`,
  `operated`, `all`).
- **Стек** агент спрашивает отдельно (можно несколько; можно пропустить — тогда
  только `common`): `1c`/`web`/`backend`/`mobile`/`desktop`/`data-ml`/
  `data-analysis`/`excel-research`/`powerbi`/`jira-confluence`/`devops`/`embedded`.
  Выбор пишется в `.best-practices.json`.
- **Возможность Jira & Confluence:** добавь `jira-confluence` поверх любого
  профиля, если проект работает с локальными Jira/Confluence через MCP. Агент
  также выбирает стек `jira-confluence`; токены и локальная конфигурация Codex
  не попадают в проект.
- ⌨ **Вручную:** `./scripts/bootstrap-new-project.sh "<путь>" "Название" software`
  (или `.ps1` на Windows).
- Дальше агент: разворачивает структуру (AGENTS/CLAUDE, README, PROJECT, INDEX,
  только подходящие профилю документы) → git init, отдельный GitHub-репозиторий,
  первый commit и push → заполнить `PROJECT`, `AGENTS → Commands`, `INDEX`.
- **Результат:** готовый стандартизированный проект с проверенными
  `.project-standard.json`, git status и wikilinks.

## Оценить существующий проект

read-only.

- **Что делает:** оценка legacy-проекта относительно стандарта и decision report:
  какой профиль ближе, возможен ли adoption in place или безопаснее новый проект.
- **Когда:** понять разрыв со стандартом без изменения файлов.
- **Где:** `Правила для нового проекта`, `--root` на целевой проект.
- 🗣 **Агенту:** *«Оцени этот проект относительно стандарта»* · `/assess-existing-project`.
- ⌨ **Вручную:** `python3 scripts/standardize_existing_project.py --root "<путь>" --json`
  (или `./scripts/standardize-existing-project.sh --root "<путь>"`).
- **Результат:** decision report (`missing_files`, `files_to_create`, рекомендация
  стратегии). Файлы не меняются.

## Стандартизировать проект

- **Что делает:** выбирает и выполняет стратегию — `adopt-in-place` (на месте) или
  `re-bootstrap-from-existing` (пересобрать по стандарту). Сначала обязательно assess.
- **Когда:** решил привести legacy-проект к стандарту.
- **Где:** `Правила для нового проекта`, `--root` на целевой проект.
- 🗣 **Агенту:** *«Предложи стратегию стандартизации и выполни выбранную»* ·
  `/standardize-existing-project`.
- ⌨ **Вручную:** план —
  `./scripts/standardize-existing-project.sh --root "<путь>" --strategy adopt-in-place --plan-adopt`;
  применить — `… --apply --fingerprint "<64-hex>" --yes`.
- **Результат:** decision report со сравнением стратегий и — по подтверждению
  fingerprint — применённая стратегия.

## Проверить и продиагностировать

read-only.

- **Что делает:** `validate-project` проверяет структуру проекта против стандарта;
  `project-doctor` — окружение компьютера.
- **Когда:** убедиться, что проект/окружение соответствуют стандарту.
- **Где:** в целевом проекте (validator) и на машине (doctor).
- 🗣 **Агенту:** *«Проверь проект валидатором и продиагностируй окружение»*.
- ⌨ **Вручную:** `./scripts/validate-project.sh --root . --kind project --profile software`;
  `./scripts/project-doctor.sh --root . --agent-mode codex`. `--report-only` — тот
  же отчёт, но exit 0.
- **Результат:** список findings и exit code; файлы не меняются.

## Сжать «мусор» проекта

- **Что делает:** Уровень 1 (скрипт) — механическая обратимая компрессия (архив
  старых `Fixed`-дефектов, сплит CHANGELOG); Уровни 2–3 (скилл) — docs и память, с
  планом до применения.
- **Когда:** проект разросся; уборка перед релизом.
- **Где:** в целевом проекте (`--root`).
- 🗣 **Агенту:** *«Сожми накопившийся мусор проекта, покажи план»* · `/compress-project`.
- ⌨ **Вручную:** отчёт — `./scripts/compress-project.sh --root .`; применить
  обратимое — `./scripts/compress-project.sh --root . --apply`.
- **Результат:** отчёт/план; по `--apply` — только обратимые операции; инварианты
  держит `validate-project`. Если изменения не закоммичены, агент явно предлагает
  сделать commit.

## Подтянуть практики Best Practices

- **Что делает:** применяет готовые практики соседней базы под стек (pull-модель):
  `common` сразу, стековые — по определению стека.
- **Когда:** определён стек проекта.
- **Где:** твой проект; skill читается из соседней `../Best Practices`.
- 🗣 **Агенту (основной маршрут):** *«Подтяни практики Best Practices под стек»* ·
  `/apply-best-practices`. Напоминание срабатывает само при определении стека.
  Отдельного скрипта нет; вручную можно читать `../Best Practices/practices/**`.
- **Результат:** `common` применён; выбор (applied/optout) в `.best-practices.json`.
  Если validation/report базы не прошёл, проект остаётся рабочим, применение
  практик откладывается без ручных outcomes, а агент даёт команду повторного запуска.

## Синхронизировать глобальные правила

read-only.

- **Что делает:** показывает состояние managed-block глобальной policy и diff;
  ничего не записывает (установка markers — отдельной подтверждаемой миграцией).
- **Когда:** проверить состояние global policy, увидеть расхождение.
- **Где:** `Правила для нового проекта`.
- 🗣 **Агенту:** *«Проверь состояние глобальных правил и покажи diff»*.
- ⌨ **Вручную:** `./scripts/sync-global-agents.sh --check`;
  `./scripts/sync-global-agents.sh --diff --report-only`.
- **Результат:** adoption-состояние (`managed_match` / `legacy_exact` /
  `unmanaged_conflict`), без записи.

## Спланировать миграцию правил

- **Что делает:** строит план миграции global/project policy с preconditions;
  `apply` — только по подтверждённому fingerprint, с backup.
- **Когда:** применить изменения policy контролируемо и обратимо.
- **Где:** `Правила для нового проекта`.
- 🗣 **Агенту:** *«Построй план миграции правил и покажи; применяй только после
  моего подтверждения fingerprint»*.
- ⌨ **Вручную:** план —
  `./scripts/plan-migration.sh --plan --target global --report-only`; применить —
  `./scripts/plan-migration.sh --apply --target global --fingerprint "<64-hex>" --yes`.
- **Результат:** план без изменений; по apply — миграция применена, путь backup
  напечатан, повторный `--check` = `managed_match`.

## Зафиксировать урок / дефект

- **Что делает:** рефлексия ошибки/приёма и запись урока в нужный артефакт:
  `DEFECTS`, `PLAYBOOK`, `AGENTS` или promotion.
- **Когда:** нашёл баг или удачный приём (сработавший ≥2 раз).
- **Где:** твой проект.
- 🗣 **Агенту:** *«Зафиксируй этот урок, занеси куда следует»* · `/reflect-and-record`.
- ⌨ **Вручную:** дописать в `docs/quality/DEFECTS.md` (баг) или
  `docs/quality/PLAYBOOK.md` (удачный приём).
- **Результат:** урок в одном основном артефакте (без дублей), при необходимости
  связан wikilink.
