Two-tier knowledge governance · ADR-0003

Работа с практиками — полный процесс

Каждый шаг: кто исполняет, чем запускается (auto / skill / script / manual / git), с каким входом и результатом, и по какому сигналу человек понимает, что настал его ход.

Тип запуска autoсам skillвызов скилла scriptкоманда manualруками gitpush / PR
Исполнитель агент·правило польз.→агент система·CI администратор
Сигнал «пора действовать» Сигнал ↓ на связке = выход прошлого шага ◆ строка = сигнал извне потока

Оглавление

  1. Поток A — находка → практика в Best Practices
  2. Поток B — затвердевание практики → правило стандарта
  3. Контур C — целостность между репозиториями
ПОТОК A

Находка → практика в Best Practices

Основной, частый путь. Всё в репозитории Best Practices.

A1Зафиксировать факт в проекте
◆ СигналОбнаружен дефект или повторившийся приём в ходе работы (внешний старт потока)
Путь по умолчанию
агент · по правилуauto Запускать нечего: агент сам пишет дефект в DEFECTS.md (правило AGENTS.md)
Путь по запросу
пользователь → агентskill «Зафиксируй этот урок, занеси в дефекты» · /reflect-and-record
ВходСбой, приём или решение из работы
→
РезультатDEFECTS.md · PLAYBOOK.md (≥2 раз) · ADR · research
A2Собрать кандидатовПользователь

→
РезультатHarvest-preview: shortlist уроков + отброшенное с причинами
Сигнал↓ harvest-preview одобрен пользователем
A3Оформить файл-кандидатПользователь
пользователь → агентscript python3 scripts/new_candidate.py … — collision-resistant ID, провенанс обязателен

→
Результатcandidates/PC-2026-<12hex>-slug.md, status new/triaged, evidence E0/E1
Сигнал↓ файл кандидата создан
A4Локальная проверкаПользователь

A6Автопроверка PRGitHub

ВходDiff Pull Request

РезультатСтатус-чек «validate»; красный блокирует merge

A7Решение: принять или отклонитьАдминистратор
администраторskill «Разбери кандидатов и прими решение» · /review-practice-candidates — дедуп, качество, обобщённость, чистота, evidence
ВходФайл кандидата + существующие practices/**
→
РезультатВердикт — ветвление ниже
✓ Accept
Создаётся practices/<раздел>/PC-*.md (E1=trial, E2/E3=accepted); кандидат → accepted + commit; wikilink в README; PR влит в main.
✕ Reject
Кандидат → rejected + причина в ## Notes. Файл не удаляется — журнал решений.
A9Доставить практику в проектыПользователь
◆ СигналПравило-напоминание в AGENTS.template срабатывает при определении стека проекта (внешний триггер)
пользователь → агентskill «Подтяни практики Best Practices под стек» · /apply-best-practices (pull-модель)
Входpractices/** + стек проекта
→
Результатcommon применён сразу; решения в .best-practices.json
Защита main (условие A5–A8): require PR + review approval + review from Code Owners (.github/CODEOWNERS) + status check validate + block force push. Настраивается владельцем один раз в GitHub Settings → Branches.
ПОТОК B

Затвердевание практики → правило стандарта

Редкий maintainer-only акт. Второй класс кандидатов — внутри new-project-rules.

B0Проверить условие входаАдминистратор
◆ СигналПрактика в BP вызрела настолько, что должна выполняться во всех проектах по умолчанию (решение администратора)
администраторmanual Проверка критериев обязательности: практика accepted, универсальна, это императив а не совет
ВходAccepted-практика BP + вывод «обязательна всем»
→
РезультатРешение начинать (иначе — оставить советом в BP)
Сигнал↓ решение о затвердевании принято
B1Gate целостности контрактаАдминистратор
администраторscript python3 scripts/check_best_practices_contract.py --best-practices-root "../Best Practices"
Входconfig/best-practices-contract.json + реальный BP
→
РезультатOK по remote/commit/hashes/status/routes/ADR — иначе стоп, сначала обновить pin
Сигнал↓ контракт-гейт зелёный
B2Оформить promotion-кандидатАдминистратор
администраторskill «Затверди эту практику как правило стандарта» · /promote-project-knowledge → promotion_candidates.py create
ВходAccepted-практика BP, точный practice path, полный source_commit
→
Результатdocs/quality/promotion-candidates/PC-*.md, status approved, задан artifact_type + target
Сигнал↓ кандидат approved
B3Материализовать в стандартАдминистратор
администраторskill «Примени этот кандидат» · /apply-promotion-candidate
ВходОдин approved кандидат по ID
→
РезультатАртефакт NPR: rule / template / test / validator / script / skill / guide
Сигнал↓ изменение внесено (тот же skill продолжает)
B4Синхронизировать репозиторийАдминистратор
администраторmanual Правки в одной задаче — тот же /apply-promotion-candidate
ВходИзменение из B3
→
РезультатОбновлены README, INDEX, docs/README, CHANGELOG, шаблоны/тесты/скилы
Сигнал↓ синхронизация завершена, проверки зелёные
B5Traceability + отправкаАдминистратор
администраторgit «Закоммить и запушь» · git push origin/main — стандарт NPR не требует PR, но агент спрашивает перед PR
ВходВнесённое изменение + пройденные checks
→
РезультатКандидат → implemented + commit; outcome в .best-practices.json; push в origin/main
Правило SSOT (B3): императив живёт в NPR (AGENTS.md, самодостаточно), обоснование остаётся в Best Practices — rationale не копируется, связывается мягкой ссылкой.
КОНТУР C

Целостность между репозиториями

Фоновая защита: pin на конкретный commit BP + хеши интерфейсных файлов, без сетевой зависимости CI.

C1Контрактный тест + Cross-repo E2EGitHub
◆ СигналЛюбой push или PR в new-project-rules (событие GitHub)
система · CIauto ci.yml → test-best-practices-contract.py + test-best-practices-e2e.py (Ubuntu / Windows / macOS)
ВходContract.json + хеши интерфейсных файлов; checkout source_commit, реальный BP practice_report.py
→
РезультатPass/fail; подтверждено, что writer NPR ↔ reader BP совместимы
C3Ежедневный сторож дрейфаGitHub · cron
◆ СигналРасписание cron 17 5 * * * (ежедневно, без участия человека)
система · CIauto bp-pin-watch.yml → check_best_practices_contract.py --check-latest
ВходReviewed pin vs текущий BP main
→
РезультатПри дрейфе — красный read-only сигнал (commit/hashes не меняет)
Сигнал↓ красный сигнал bp-pin-watch
C4Обновление pinАдминистратор
администраторgit Вручную, отдельным reviewable PR (5 шагов из контракт-доки) — не смешивать с реализацией правила
ВходНовый BP main, accepted-практика с достаточной evidence
→
РезультатОбновлены commit / hashes / route / ADR expectations
Две решающие точки — одна роль администратора: A7 — пустить ли опыт в базу практик (accept/reject); B0/B2 — сделать ли практику обязательным правилом. Прежде различались «мейнтейнер» и «администратор» — теперь это единая роль ведения базы и стандарта.
