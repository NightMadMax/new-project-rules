Работа с проектами — карта действий пользователя

Что можно делать с проектами по стандарту. Каждое действие расписано одинаково: что делает, когда, где, два маршрута (🗣 агенту / ⌨ вручную) и результат.

Почему не один сквозной workflow. Линейная модель «шаг за шагом» подходит только для процессов (создание проекта). «Оценить», «стандартизировать», «сжать» — самостоятельные действия. Поэтому это карта равноправных действий. Почти всё запускается из папки Правила для нового проекта; действия над своим проектом — из его папки.

Оглавление

  1. Создать новый проект
  2. Оценить существующий проект read-only
  3. Стандартизировать проект
  4. Проверить и продиагностировать read-only
  5. Сжать «мусор» проекта
  6. Подтянуть практики Best Practices
  7. Синхронизировать глобальные правила read-only
  8. Спланировать миграцию правил
  9. Зафиксировать урок / дефект
1

Создать новый проект

Что делаетРазворачивает новый проект по стандарту: отдельный git-репозиторий внутри общего vault, стартовые документы, отдельный GitHub-репозиторий.
КогдаНачинаешь новый проект.
ГдеПравила для нового проекта — bootstrap использует её commit как проверяемый source_commit.
🗣 агенту
«Создай новый проект „Название“ в папке <путь>, профиль software, стек web и backend, приватный GitHub-репозиторий» · /create-new-project. Профиль по умолчанию software. Стек агент спрашивает отдельно (можно несколько или пропустить → только common): 1c/web/backend/mobile/desktop/data-ml/data-analysis/excel-research/powerbi/jira-confluence/devops/embedded → пишется в .best-practices.json.
⌨ вручную
./scripts/bootstrap-new-project.sh "<путь>" "Название" software(или .ps1 на Windows).
  1. Агент разворачивает структуру: папка как отдельный git-репозиторий; AGENTS.md/CLAUDE.md, README, PROJECT, INDEX; только подходящие профилю документы.
  2. Публикация: git init → отдельный GitHub-репозиторий → первый commit и push (local-only — без GitHub).
  3. Заполнить стартовые документы: PROJECT (цель/scope/non-goals), AGENTS → Commands (только проверенные команды), INDEX.
Результатготовый стандартизированный проект с проверенными .project-standard.json, git status и wikilinks.
↑ к оглавлению

Оценить существующий проект

read-only
Что делаетRead-only оценка legacy-проекта относительно стандарта и decision report: какой профиль ближе, можно ли adoption in place или безопаснее новый проект.
КогдаНужно понять разрыв со стандартом без изменения файлов.
ГдеПравила для нового проекта, --root указывает на целевой проект.
🗣 агенту
«Оцени этот проект относительно стандарта» · /assess-existing-project
⌨ вручную
python3 scripts/standardize_existing_project.py --root "<путь>" --json(или ./scripts/standardize-existing-project.sh --root "<путь>").
Результатdecision report (missing_files, files_to_create, рекомендация стратегии). Файлы не меняются.
↑ к оглавлению
3

Стандартизировать проект

Что делаетВыбирает и выполняет стратегию: adopt-in-place (привести на месте) или re-bootstrap-from-existing (пересобрать по стандарту на основе существующего). Сначала обязательно assess.
КогдаРешил привести legacy-проект к стандарту.
ГдеПравила для нового проекта, --root на целевой проект.
🗣 агенту
«Предложи стратегию стандартизации и выполни выбранную» · /standardize-existing-project
⌨ вручную
План:./scripts/standardize-existing-project.sh --root "<путь>" \ --strategy adopt-in-place --plan-adoptПрименить тем же fingerprint:… --apply --fingerprint "<64-hex>" --yes
Результатdecision report со сравнением стратегий и — по подтверждению fingerprint — применённая стратегия (безопасные автоматические действия).
↑ к оглавлению

Проверить и продиагностировать

read-only
Что делаетvalidate-project проверяет структуру проекта против стандарта; project-doctor — окружение компьютера (Git, gh, инструменты, agent mode).
КогдаУбедиться, что проект/окружение соответствуют стандарту и готовы.
ГдеВ целевом проекте (validator) и на машине (doctor).
🗣 агенту
«Проверь проект валидатором и продиагностируй окружение»
⌨ вручную
./scripts/validate-project.sh --root . --kind project --profile software ./scripts/project-doctor.sh --root . --agent-mode codex--report-only — тот же отчёт, но exit 0.
Результатсписок findings и exit code; файлы не меняются.
↑ к оглавлению
5

Сжать «мусор» проекта

Что делаетУровень 1 (скрипт) — механическая обратимая компрессия: архив старых Fixed-дефектов, сплит разросшегося CHANGELOG. Уровни 2–3 (скилл) — docs и память агентов, с планом до применения.
КогдаПроект разросся; периодическая уборка перед релизом.
ГдеВ целевом проекте (--root).
🗣 агенту
«Сожми накопившийся мусор проекта, покажи план» · /compress-project
⌨ вручную
Отчёт (read-only):./scripts/compress-project.sh --root .Применить обратимое:./scripts/compress-project.sh --root . --apply
Результатотчёт/план; по --apply — только обратимые операции (архив, сплит), инварианты держит validate-project.
↑ к оглавлению
6

Подтянуть практики Best Practices

Что делаетПрименяет готовые практики соседней базы под стек проекта (pull-модель): common сразу, стековые — по определению стека.
КогдаОпределён стек проекта; хочешь применить готовые практики.
Гдетвой проект; skill читается из соседней ../Best Practices.
🗣 агенту (основной маршрут)
«Подтяни практики Best Practices под стек» · /apply-best-practices. Напоминание срабатывает само при определении стека. Отдельного скрипта нет — доставка идёт через skill; вручную можно читать ../Best Practices/practices/**. Если validation/report падает, проект сохраняется, применение откладывается без ручных outcomes, агент показывает ошибку и команду повтора.
Результатcommon применён; выбор (applied/optout) зафиксирован в .best-practices.json.
↑ к оглавлению

Синхронизировать глобальные правила

read-only
Что делаетПоказывает состояние managed-block глобальной policy на машине и diff. Ничего не записывает — установка markers идёт отдельной подтверждаемой миграцией.
КогдаПроверить состояние global policy, увидеть расхождение.
ГдеПравила для нового проекта.
🗣 агенту
«Проверь состояние глобальных правил и покажи diff»
⌨ вручную
./scripts/sync-global-agents.sh --check ./scripts/sync-global-agents.sh --diff --report-only
Результатadoption-состояние (managed_match / legacy_exact / unmanaged_conflict), без записи.
↑ к оглавлению
8

Спланировать миграцию правил

Что делаетСтроит план миграции global/project policy с preconditions; apply — только по подтверждённому fingerprint, с backup.
КогдаПрименить изменения policy контролируемо и обратимо.
ГдеПравила для нового проекта.
🗣 агенту
«Построй план миграции правил и покажи; применяй только после моего подтверждения fingerprint»
⌨ вручную
План (ничего не меняет):./scripts/plan-migration.sh --plan --target global --report-onlyПрименить:./scripts/plan-migration.sh --apply --target global \ --fingerprint "<64-hex>" --yes
Результатплан без изменений; по apply — миграция применена, путь backup напечатан, повторный --check = managed_match.
↑ к оглавлению
9

Зафиксировать урок / дефект

Что делаетРефлексия ошибки/приёма и запись обобщённого урока в нужный артефакт: DEFECTS, PLAYBOOK, AGENTS или promotion.
КогдаНашёл баг или удачный приём (сработавший ≥2 раз).
Гдетвой проект.
🗣 агенту
«Зафиксируй этот урок, занеси куда следует» · /reflect-and-record
⌨ вручную
Дописать запись в docs/quality/DEFECTS.md (баг) или docs/quality/PLAYBOOK.md (удачный приём).
Результатурок зафиксирован в одном основном артефакте (без дублей), при необходимости связан wikilink.
↑ к оглавлению
