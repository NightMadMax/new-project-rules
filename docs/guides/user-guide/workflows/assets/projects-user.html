Работа с проектами — карта действий пользователя

Что можно делать с проектами по стандарту. «Создать проект» — это процесс (шаги ниже); остальное — независимые действия-возможности, каждое со своим триггером. Выбирай нужное.

Почему не один workflow. Линейная модель «шаг за шагом» подходит только для процессов (создание проекта, настройка компьютера). «Оценить», «стандартизировать», «валидировать», «сжать» — самостоятельные действия, а не одна последовательность. Поэтому здесь — карта: у каждого действия свой «когда» и «чем запускается». Почти всё живёт в папке Правила для нового проекта.

Процесс

Создать новый проект

Где: открой агента в папке Правила для нового проекта (bootstrap использует её commit как проверяемый source_commit).

Запросить создание · два маршрута
🗣 Агенту: «Создай новый проект „Название“ в папке <путь>, профиль software, приватный GitHub-репозиторий» · /create-new-project. Профиль по умолчанию — software (есть minimal, operated, all).
⌨ Вручную: ./scripts/bootstrap-new-project.sh "<путь>" "Название" software (или .ps1 на Windows).
Агент разворачивает структуру
Создаёт папку внутри общего vault как отдельный git-репозиторий; заполняет AGENTS.md/CLAUDE.md, README, PROJECT, INDEX; добавляет только подходящие профилю документы.
Публикация и проверка
git init → отдельный GitHub-репозиторий → первый commit и push (для local-only — без GitHub). Проверяет .project-standard.json, bootstrap, git status, wikilinks.
Заполнить стартовые документы
PROJECT: цель/scope/non-goals; AGENTS → Commands: только реально проверенные команды; INDEX: только существующие файлы. Есть код — заполнить install/build/test/lint/run.
Каталог

Остальные действия с проектами

Независимые возможности. У каждой — когда применять и чем запускается.

🔎Оценить существующий проектread-only
Когда: есть legacy-проект, хочешь понять разрыв со стандартом без изменения файлов.
Запуск«Оцени этот проект относительно стандарта» · /assess-existing-project → decision report
🧭Стандартизировать проект
Когда: решил привести legacy к стандарту — на месте или пересобрать по шаблону.
Запуск«Предложи стратегию стандартизации» · /standardize-existing-project
✅Проверить и продиагностироватьread-only
Когда: убедиться, что проект/окружение соответствуют стандарту.
Запускpython3 scripts/validate-project.py · ./scripts/project-doctor.sh
🗜️Сжать «мусор» проекта
Когда: проект разросся; периодическая уборка перед релизом.
Запуск«Сожми накопившийся мусор проекта» · /compress-project
📥Подтянуть практики Best Practices
Когда: определён стек; хочешь применить готовые практики. Где: твой проект.
Запуск«Подтяни практики под стек» · /apply-best-practices
🔄Синхронизировать глобальные правилаread-only check
Когда: проверить состояние global policy на машине, увидеть diff.
Запуск./scripts/sync-global-agents.sh --check · --diff --report-only
🧱Спланировать миграцию правил
Когда: нужно применить изменения global/project policy контролируемо.
Запуск./scripts/plan-migration.sh --plan --report-only, apply — по fingerprint
📝Зафиксировать урок / дефект
Когда: нашёл баг или удачный приём. Где: твой проект.
Запуск«Зафиксируй урок» · /reflect-and-record → DEFECTS / PLAYBOOK
Часть серии workflow пользователя. Каждое действие при необходимости раскрывается в собственный детальный workflow. Справочники — в docs/guides.
