# Документирование процесса как workflow-отчёта

Задача skill — создать или обновить визуальный workflow-отчёт процесса и держать
его синхронным с реальностью. Отчёты живут в
`docs/guides/user-guide/workflows/`: каждая страница — Markdown (source of truth,
в графе Obsidian) с приложенным самодостаточным HTML в `assets/` тем же basename.

## Когда применять

- Добавить отчёт по новому процессу.
- Обновить отчёт после изменения процесса: новый/переименованный skill, новая
  команда скрипта, изменённый порядок шагов, новая роль, перенос файлов.
- Привести рассинхронившийся отчёт в соответствие с текущими скиллами и скриптами.

Не использовать для правки содержания процесса по существу — только для его
описания.

## Подготовить модель

1. Полностью прочитать [[docs/guides/user-guide/README|модель]] и
   [[docs/guides/user-guide/workflows/README|каталог workflow]] — они имеют
   приоритет над этим workflow.
2. Взять существующий отчёт как образец вёрстки: HTML в
   `docs/guides/user-guide/workflows/assets/` (rail, токен-палитра, светлая/тёмная
   тема, моноширинный шрифт для идентификаторов кода).
3. Сверить факты с источником: реальные `.agents/skills/`, `scripts/`,
   `.github/workflows/` и гайды. Не выдумывать команды и триггеры — проверять.

## Модель отчёта

Каждый шаг кодирует поля:

1. **Действие** — краткий заголовок (+ step ID для полного процесса).
2. **Исполнитель** — категория (цветной чип): агент·по правилу / пользователь→
   агент / система·CI / администратор. Роли мейнтейнера и администратора
   объединены в администратора.
3. **Запуск** — тип + чем именно: `auto` / `skill` (фраза агенту + `/имя`) /
   `script` (точная команда) / `manual` / `git` (push/PR).
4. **Где** (для ролевых отчётов) — в какой папке-репозитории открыть агента;
   скилы привязаны к репозиторию. «Что сделать» — нумерованные под-шаги: открой
   агента в папке X → напиши фразу / вызови `/skill`.
5. **Вход → Результат**.
6. **Сигнал** — как человек понимает, что настал его ход. Гибрид: **Д2** (подпись
   на связке между шагами), когда сигнал = выход предыдущего шага; **Д1** (строка
   «◆ Сигнал» на карточке) для сигналов извне потока (старт, cron, решение
   администратора, правило-напоминание).

Раздельные пути: где действие происходит и само (агент по правилу), и по запросу
пользователя — показывать двумя путями в одной карточке.

**Два маршрута исполнения.** Где шаг можно сделать и попросив агента, и запустив
скрипт руками — показывать оба маршрута явно и раздельно: **🗣 Агенту** —
конкретная фраза, которой просишь сделать шаг (+ `/skill`, где есть), и
**⌨ Вручную** — точная команда/скрипт. Если маршрут только один (например,
Obsidian «Open folder as vault» — только руками; агент не может), показывать его
и явно пометить, что второго нет.

## Области охвата и наименование

- **Полный процесс** — все исполнители.
- **Отчёты по ролям** — только шаги одной роли, в логике «что сделать, чтобы
  добиться результата»; чужие/автоматические шаги — пассивные маркеры («⚙ само»,
  «⏳ ожидание»).
- Если процесс **не линейный** (набор независимых действий, а не одна
  последовательность) — оформлять **картой действий** (каталог карточек
  «действие → когда → Где → чем запускается»), а не rail. Линейный rail — только
  для настоящих процессов.

Заголовок каждой страницы — по схеме **`<действие> — <охват/роль>`**
(например «Работа с практиками — workflow пользователя»).

## Создать или обновить

1. Markdown — source of truth; HTML в `assets/` — тем же basename. Оба
   обновлять в одной задаче, чтобы не расходились.
2. Для нового процесса: создать `<process>-full.md` и нужные ролевые
   `<process>-<роль>.md` + HTML; при нелинейности — одну карту действий.
3. Для обновления: найти затронутый отчёт, поправить и Markdown, и HTML, сверив
   с текущими скиллами/скриптами.

## Синхронизировать репозиторий

1. Обновить в одной задаче: [[docs/guides/user-guide/workflows/README|каталог]],
   [[INDEX]], `docs/README.md` и `CHANGELOG.md`, если изменились состав, названия
   или назначение отчётов.
2. Соблюдать [[AGENTS]] и правило Markdown-workflow: викилинки, без дублей.

## Проверить

1. Запустить `python3 scripts/validate-project.py` — 0 ошибок.
2. Проверить, что нет битых викилинков и устаревших путей.
3. HTML самодостаточен (инлайн CSS, без внешних ссылок), светлая/тёмная тема
   работают.
4. Напомнить: опубликованный артефакт — снимок на момент публикации; после правки
   HTML ссылку нужно перезаливать отдельно. Файлы в репозитории всегда актуальны.

## Отчёт

1. Какой процесс задокументирован/обновлён и какие файлы затронуты.
2. Какие проверки пройдены.
3. Что осталось (например, перезалить артефакт или синхронизировать git).
