# Компрессия проекта

Периодически снимать накопления, из-за которых проект деградирует как контекст:
разросшиеся журналы, наслоившиеся одноразовые research/audit, устаревшие
промо-кандидаты, дрейф навигации, технический мусор и раздутую память агентов.
Провенанс никогда не удаляется — только перемещается в архив. Всё под git и
проверяемо через `git diff`.

Правило безопасности: сначала отчёт, затем показ плана, затем применение.
Ничего необратимого без явного подтверждения пользователя.

## Уровень 1 — механический (скрипт)

1. Запустить в режиме отчёта (без `--apply`):
   `python3 scripts/compress-project.py --root .` и
   `python3 scripts/validate-project.py --root . --doctor --report-only`.
2. Показать пользователю запланированные действия и notices как есть.
3. По подтверждению выполнить `scripts/compress-project.py --root . --apply`.
   Скрипт делает только обратимое: перенос `Fixed`-дефектов старше порога в
   `docs/quality/DEFECTS_ARCHIVE.md`, сплит переросшего `CHANGELOG.md` в
   `CHANGELOG_ARCHIVE.md` (Unreleased + последние релизы остаются), удаление
   регенерируемого мусора (`__pycache__`, `.DS_Store`, пустой `.trash`).
4. Этот шаг автоматизирует ручную «Консолидацию журналов» из
   `reflect-and-record`; не выполнять её вручную, если скрипт уже отработал.

## Уровень 2 — консолидация docs (суждение, по умолчанию в скилле)

Скрипт не трогает содержательные файлы — это делает агент, с показом плана до
применения и с сохранением исходников в архив, а не удалением.

1. Найти наслоившиеся/устаревшие файлы в `docs/research/` и `docs/reviews/`
   (например, несколько `PROJECT_AUDIT_*`): определить, что чем вытеснено;
   слить в один актуальный, устаревшие пометить `status: superseded` и
   перенести, сохранив исходный текст.
2. В `docs/quality/promotion-candidates/` проверить записи `implemented`/`rejected`
   в архивную секцию; открытые кандидаты не трогать.
3. Починить дрейф навигации, найденный валидатором: битые wikilinks, ссылки
   `INDEX.md`/`docs/README.md` на перемещённые или удалённые файлы.
4. Показать сводный план (что сливается, что архивируется, что переименовано) и
   применять только после подтверждения.

## Уровень 3 — память агентов (opt-in, `--include-memory`)

Память живёт **вне git** (`.codex/memories`, `.claude/projects`,
`.claude/memory` — см. `RAW_MEMORY_PATHS` в валидаторе), поэтому операции
необратимы: нет `git revert`, архивировать в репозиторий нельзя. Выполнять
только по явному запросу при оптимизации проекта и строго по шагам:

1. **Отчёт**: показать, что раздулось, где дубликаты фактов, где `MEMORY.md`
   ссылается на несуществующие файлы.
2. **Показ дублей**: перечислить пары/группы дублирующих фактов пользователю.
3. **Подтверждение**, затем только: слияние подтверждённых дублей в один файл и
   починка индекса `MEMORY.md`.
4. **Никакого автоудаления** файлов-фактов. Кандидатов на удаление только
   показать пользователю для ручного решения. Память не коммитить.

## Завершение

1. Обновить `INDEX.md` и `docs/README.md`, если файлы перемещены, слиты или
   переименованы.
2. Commit и push выполнять только когда это разрешено запросом пользователя и
   repository workflow. Если изменения остались незакоммиченными, в итоговом
   отчёте явно предложить пользователю сделать commit; не оставлять этот шаг
   неозвученным. Указать, что перенесено и куда, и что осталось на ручное решение.
3. При первом успешном прогоне зафиксировать паттерн в `docs/quality/PLAYBOOK.md`
   (проверен минимум дважды).
